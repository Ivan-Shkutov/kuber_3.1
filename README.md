### РЕШЕНИЕ:

### Расчёт требуемых ресурсов для Kubernetes-кластера:

1. Исходные данные

Приложение состоит из:

  - Backend (10 копий)

  - Frontend (5 копий)

  - База данных (3 копии, отказоустойчивая)

  - Кеш (3 копии, отказоустойчивый)

Приложение планируется упаковать в Helm chart для деплоя в разные окружения (например: dev / stage / prod).

2. Расчёт ресурсов по компонентам
   
    2.1 База данных (3 копии)

        4 ГБ RAM × 3 = 12 ГБ
        1 CPU × 3 = 3 CPU

    2.2 Кеш (3 копии)

        4 ГБ RAM × 3 = 12 ГБ
        1 CPU × 3 = 3 CPU

    2.3 Backend (10 копий)

        600 МБ × 10 = 6000 МБ ≈ 6 ГБ RAM
        1 CPU × 10 = 10 CPU

    2.4 Frontend (5 копий)

        50 МБ × 5 = 250 МБ ≈ 0.25 ГБ RAM
        0.2 CPU × 5 = 1 CPU


3. Суммарные ресурсы (без запаса)

        Компонент	        RAM          	CPU
        БД	              12 ГБ        	3
        Кеш	              12 ГБ        	3
        Backend	          6 ГБ      	  10
        Frontend	        0.25 ГБ  	    1
        ИТОГО	            30.25 ГБ  	  17 CPU

4. Добавим служебные ресурсы Kubernetes

Каждая worker-нода должна иметь запас на:

    kubelet
    container runtime
    kube-proxy
    CNI
    логирование
    метрики

5. Stateful-ноды (БД + кеш)

Суммарно:

    12 + 12 = 24 ГБ RAM
    3 + 3 = 6 CPU

Добавим запас на отказ одной ноды (N+1). Минимально разумная схема — 3 ноды

Если распределить по 2 stateful-пода на ноду, то каждая нода:

    8 ГБ RAM под workload
    2 CPU
    +1 ГБ системных

Итого по stateful:

    3 ноды × 16 ГБ = 48 ГБ RAM
    3 ноды × 4 CPU = 12 CPU

Почему:

    есть запас по памяти
    возможна деградация при падении одной ноды
    остаётся место для перераспределения pod’ов

6. Stateless-ноды (backend + frontend)

Суммарно:

    6.25 ГБ RAM
    11 CPU

CPU — основной лимитирующий фактор.

Если взять ноды (3 ноды), то каждая:

    8 CPU
    16 ГБ RAM
    3 × 8 CPU = 24 CPU

7. Финальная конфигурация кластера

    
Stateful pool

    3 ноды
    4 CPU
    16 ГБ RAM

Stateless pool

    3 ноды
    8 CPU
    16 ГБ RAM

ИТОГО

    Тип        	  Количество     	CPU        	RAM
    Stateful	    3              	4          	16 ГБ
    Stateless	    3              	8          	16 ГБ
    Всего нод    	6              	36 CPU    	96 ГБ RAM



8. Что обеспечивает такая схема

  - Отказоустойчивость (переживает падение одной ноды в каждом пуле)

  - Распределение stateful-нагрузки

  - Изоляцию БД и кеша от backend

  - Возможность горизонтального масштабирования

  - Возможность использования nodeSelector / taints / tolerations


9. Архитектурные решения

Для БД и кеша:
        
    StatefulSet
    PersistentVolumeClaim
    PodDisruptionBudget
    AntiAffinity
    nodeSelector / taints

Для backend и frontend:

    Deployment
    HorizontalPodAutoscaler
    PodDisruptionBudget
    readiness / liveness probes

10. Вывод

Кластер из 6 worker-нод:

    обеспечивает отказоустойчивость (N+1)
    разделяет stateful и stateless нагрузку
    имеет достаточный запас по CPU и RAM
    готов к масштабированию
    подходит для деплоя через Helm в разные окружения
